TriBench

A Formalized Benchmark for GPU Code Generation

Reproducible. • Extensible. • Data-Driven.

The Motivation

The Status Quo is Chaotic

Evaluating Triton kernels and AI-generated GPU code today is fraught with friction:

  • Unstandardized Scripts: Every repo has a bespoke benchmark.py.
  • JIT Pollution: Warmup and JIT compilation overheads often leak into latency timings.
  • Missing Metrics: Relying purely on average latency hides critical tail-latency (p95) issues.
  • Code Generation Gap: LLMs are generating Triton Kernels, but we lack a strictly formalized, data-driven suite to evaluate them fairly.

The Benchmark Landscape

TritonBench

Harvested massive volume of real-world operators. Great scale, but monolithic interface.

KernelBench / GEAK

Focuses on LLM writing kernels, but architectures are often tightly coupled to specific LLM pipelines or hardware (AMD).

TriBench

Highly Formalized & Decoupled. Evaluates kernels strictly through a data-driven configuration, isolating compilation from measurement for perfect A/B testing.

🧠 Design Architecture

A decoupled, data-driven philosophy.

Zero Framework Mod

Adding a new kernel requires zero changes to the core framework. Simply drop a self-contained folder into kernels/.

Data-Driven Meta

Each kernel defines its entrypoints, test shapes, and benchmark parameters locally via a meta.json file.


tribench/        # Core CLI
kernels/         # Self-contained
  ├── rope/
  │   ├── meta.json         
  │   ├── reference.py      
  │   ├── triton_impl.py  
  │   └── bench_entry.py    
scripts/         # Automations
tests/           # Unit tests

📊 Metrics That Matter

Meaningful kernel evaluation goes beyond average latency.

  • p50 / p95 Latency:
    Reveals tail-latency issues (e.g. from bad thread scheduling or memory bank conflicts) that mean-latency hides.
  • Hardware Utilization (TFLOPS):
    Allows roofline analysis to easily identify if a kernel is compute-bound.
  • Memory Bandwidth (GB/s):
    Critical for IO-bound operators (e.g., LayerNorm, SwiGLU) to track efficiency.

🚀 Accelerating Development

Eradicate the boilerplate of writing PyTorch vs. Triton integration tests.

1. Validate Shapes

tribench validate-meta

Ensures `meta.json` configurations are sound.

2. Test Correctness

tribench test --kernel all

Checks max diff vs PyTorch reference.

3. Benchmark Variants

tribench run --kernel rope --dtype fp16

Perfectly isolated A/B testing of your core Tritons against experimental sequential baselines.

🧪 Proving Correctness

Before benchmarking, kernels must be proven correct. TriBench automates this rigorously:

  • PyTorch Reference Baseline:
    Every kernel mandates a reference.py implementing the exact math in pure PyTorch.
  • Shape & Dtype Sweep:
    meta.json defines the test grid. TriBench automatically sweeps over all sizes and precisions (fp16, bf16, fp32).
  • Strict Tolerance Checking:
    Compares the Triton kernel's output and gradients against the PyTorch reference, ensuring maximum difference (max_diff) is within acceptable bounds.

🧩 The Power of Composition

Why do we need a benchmark framework, and not just a single script?

Variant A/B Testing

Triton performance is incredibly sensitive to block sizes and memory layouts. TriBench allows you to declare multiple variants in JSON to test hypotheses instantly under identical conditions, without touching the bench script.

Fused vs. Sequential

Is your complex fused kernel actually faster than calling two basic PyTorch operators sequentially? TriBench lets you register the sequential baseline as a variant to prove your optimization's worth.
