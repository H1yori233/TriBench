TriBench

A Reproducible Benchmark Suite for Triton Kernels

Why Benchmark?

Comparability & Reproducibility

GPU kernel timings are extremely sensitive — JIT compilation, warmup strategy, repetition count, and cache state all affect results. Triton's do_bench exposes warmup/rep/quantiles precisely because the measurement method is itself part of the benchmark. Without standardization, numbers across projects are simply incomparable.

Maintainability & Extensibility

A benchmark suite is a long-term asset, not a one-off script. It must absorb new kernels, shapes, dtypes, and backends over time without breaking existing results or readability. KernelBench's streamlined "correctness → timing → aggregation" pipeline shows this principle in action.

Diagnostic Value

Reporting "time (ms)" alone rarely explains bottlenecks. NVBench treats throughput and bandwidth as first-class outputs and supports cold/batch modes to control cache and noise — the same motivation behind TriBench reporting TFLOPS and GB/s.

Course to Engineering

Collect→ Reproduce→ Formalize→ Aggregate→ Publish

The Problem

Triton lets you write a kernel → JIT-compile → execute on GPU. But each repo ships its own ad-hoc benchmark.py with inconsistent warmup, timing, and reporting. Cross-project comparison is nearly impossible.

The course requires formalizing kernels into a suite, not just single-point optimizations. We need a shared skeleton that enforces correctness checking, fair timing, and interpretable metrics across all kernels.

TriBench's Role

Provides the engineering skeleton for the Formalize and Aggregate stages:

  • Unified meta.json schema per kernel
  • Automatic correctness gating before timing
  • Standardized warmup / rep / quantile measurement
  • Derived metrics (TFLOPS, GB/s) for interpretability
  • JSON + Markdown result logging with environment snapshot

→ Any community kernel can be turned from a loose script into a publishable, reproducible benchmark entry.

Core Design

📦 Self-Contained

Each kernel dir ships meta.json + reference.py + triton_impl.py. Adding a kernel = adding a folder — zero framework changes.

🔌 Lazy Import

Kernels load on demand. A crash in one never affects the global run — critical for a growing 28-kernel suite.

⚖️ Fair Timing

Warmup, rep, multiple timer backends. Compilation is isolated from measurement for clean A/B testing.

🔮 Future-Ready

Backend concept exists in meta schema — prepared for ROCm/HIP expansion without redesign.

Benchmark Pipeline

📄 Parse Meta→ 🎲 Gen Inputs→ ✅ Correctness Gate→ ⏱ Timing Loop→ 📊 Derive Metrics→ 💾 Save Results

Why correctness first?

Benchmarking a numerically wrong kernel is meaningless. By gating on correctness, every timing result is guaranteed to represent a valid implementation. The gate checks max-diff of forward output and gradients against the PyTorch reference.

What the timing loop does

Runs configurable warmup iterations (discarded), then repeated timed executions. Returns full latency distribution — min, max, mean, p50, p95. Supports 3 backends: do_bench, do_bench_cudagraph, and raw CUDA events.

What gets saved

  • Latency quantiles (p50 / p95)
  • TFLOPS and GB/s
  • Environment: torch/triton/CUDA/GPU/driver
  • Git commit hash + random seed
  • Full CLI command line

→ JSON (machine) + Markdown (human)

Timer Backends

do_bench

warmuprepquantiles

Standard approach. Configurable warmup/repetitions, returns min/max/mean or full quantile distribution via return_mode.

do_bench_cudagraph

low noiseshort kernels

CUDA Graphs eliminate Python scheduling and kernel launch overhead. Ideal for sub-100μs kernels where launch cost dominates.

CUDA Events

low-levelmanual sync

Direct GPU timestamps. Maximum control — you handle warmup and synchronization. Consistent with microbenchmark engineering practices.

Latency Distribution

← p50 p95 →

Mean latency hides tail spikes from scheduling jitter, resource contention, or memory bank conflicts. Quantile reporting (p50 + p95) surfaces them — giving a far more honest picture.

Throughput

📐 TFLOPS — Compute Throughput

Maps raw latency to compute utilization. Enables roofline analysis to determine if a kernel is compute-bound.

e.g. 85 TFLOPS / 312 peak = 27% utilization

Useful for compute-heavy ops: matmul, flash_attn, fused_linear_ce

📊 GB/s — Memory Bandwidth

Critical for memory-bound operators where data movement is the bottleneck, not arithmetic.

e.g. 1200 GB/s / 2039 peak = 59% bandwidth

Useful for IO-bound ops: layer_norm, rms_norm, swiglu, silu

External validation: NVBench treats throughput and bandwidth as first-class outputs with explicit bandwidth utilization. TriBench adopts the same principle — make results interpretable, not just fast.

Excluded Metrics

Not "we don't know" — rather "we know but choose not to," for clear engineering reasons.

Speedup (Relative Metrics)

KernelBench uses fast_p (correctness + speedup) for scoring. But speedup requires a strict, stable baseline — PyTorch eager? torch.compile? Once the baseline changes, cross-environment comparability breaks.

Decision: TriBench focuses on absolute kernel-level profiling. Speedup can be a future extension with a clearly pinned baseline.

future extension

Hardware-Level Profiling

SM occupancy, warp stall reasons, L2 hit rate, tensor core util, DRAM throughput. These require Nsight Compute, bringing three costs:

  • Profiling perturbs the measured object
  • Strong coupling to toolchain/driver versions
  • High interpretation barrier
advanced diagnostic mode

Energy / Power Metrics

Joules/op, Watts, thermal throttling. Requires hardware sensors and a clock-locked environment. NVBench supports "lock GPU clocks (Volta+)" for this — reflecting the need for additional system control planes.

Decision: Without a unified testbed, not viable as a default KPI for a course project.

needs controlled env

What variants means in meta.json

A single kernel pack can host multiple implementations. TriBench runs all of them under identical case / dtype / layout / seed / timer conditions — guaranteeing apples-to-apples comparison with zero extra configuration.

Why this matters for the course

When collecting community kernels, you inevitably accumulate reference implementations, optimized versions, and experimental variants. The variants field carries the need to formalize + compare — with zero configuration overhead per new variant.

⚡ Case: Add+SiLU Fused

One kernel does add + silu in a single pass.

  • 1 memory round-trip
  • 1 kernel launch
  • Minimal memory traffic

🔗 Case: Sequential Baseline

Two ops: torch.add → F.silu.

  • 2 memory round-trips
  • 2 kernel launches
  • Higher memory traffic

Both in one meta.json → comparison is zero-cost and cheat-proof

Correctness Validation

1

PyTorch Reference Baseline

Every kernel mandates a reference.py implementing the exact math in pure PyTorch. This is the ground truth for numerical correctness.

2

Shape & Dtype Sweep

meta.json defines the full test grid. TriBench automatically sweeps over all specified sizes and precisions (fp16, bf16, fp32) — no manual test writing needed.

3

Strict Tolerance Checking

Compares Triton output and gradients against the reference. Maximum difference (max_diff) must stay within dtype-appropriate bounds.

4

Gate Before Benchmark

Only kernels passing correctness are admitted to the timing stage. This prevents publishing meaningless performance numbers for broken implementations.

Example: RMS Norm

$ tribench test --kernel rms_norm

Testing rms_norm...
  fp32  (2048, 768)    max_diff=1.2e-6 ✅
  fp16  (2048, 768)    max_diff=3.1e-4 ✅
  bf16  (2048, 768)    max_diff=4.7e-4 ✅
  fp16  (4096, 1024)   max_diff=2.8e-4 ✅
  fp16  (8192, 4096)   max_diff=3.5e-4 ✅

All 5 cases passed.

Correctness is non-negotiable — it's the foundation that makes all downstream metrics trustworthy.
