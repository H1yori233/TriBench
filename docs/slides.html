TriBench

A Standard for Triton

Reproducible. • Extensible. • Fast.

The Motivation

Evaluating Triton kernels today is chaotic. It involves unstandardized scripts, manual data collection, and questionable reproducibility.

TriBench solves this.

It provides a standardized environment for testing correctness, profiling latency, and comparing kernel optimizations (like fused vs. sequential) with perfect reproducibility.

✨ Key Features

Standardized CLI

A unified command-line tool to list, test, and benchmark all registered kernels without writing custom harnesses.

Fair Comparison

Compile and warmup phases are strictly isolated from the measurement loop to absolutely eliminate JIT overhead.

Kernel Composition

Easily swap and compare multiple kernel variants alongside your main implementation.

Rich Metrics

Strict random seed control, p50/p95 latency, hardware TFLOPS, and memory bandwidth (GB/s).

🧠 Design Philosophy

A decoupled, data-driven architecture.

Zero Framework Modification

Adding a new kernel requires zero changes to the core framework. Simply drop a self-contained folder into kernels/.

Data-Driven Configuration

Each kernel defines its entrypoints, test shapes, and benchmark parameters locally via a meta.json file.

Project Structure


tribench/          # Core framework logic and CLI
kernels/           # Self-contained kernels
  ├── rope/
  │   ├── meta.json         # Config & test cases
  │   ├── reference.py      # PyTorch baseline
 triton_impl.py    # Triton logic  │   ├─
  │   └── bench_entry.py    # Custom benchmarks
scripts/           # Automations
tests/             # Framework unit tests

🚀 Commands

tribench list

List all available kernels.

tribench validate-meta

Verify all meta.json files.

Run checks against PyTorch:

tribench test --kernel all

Execute Benchmarks:

tribench run --kernel rope --dtype fp16

🛠 Extending TriBench

  1. Create dir under kernels/<your_kernel>/
  2. Define shapes in meta.json
  3. PyTorch baseline in reference.py
  4. Triton logic in triton_impl.py
  5. Verify via tribench test

Variant Evaluation

To compare different implementations (like a fused vs sequential approach), declare them in the variants block:


"entrypoints": {
    "triton": "fused.py:run",
    "variants": { 
        "sequential": "baseline.py:run",
        "triton_v2": "experimental.py:run"
    }
}
